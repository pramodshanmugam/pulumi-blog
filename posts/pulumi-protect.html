How Pulumi protect stops faulty deletes

The question every platform team asks before approving Pulumi — and the controls that make “faulty delete” a blocked operation instead of an outage.

When I tried to get Pulumi approved, the loudest concern was not licensing or language choice. It was faulty deletes: if infrastructure is real code, what stops a rename, a bad refactor, or a mistaken stack update from destroying a database, a disk, or a cluster?

That fear is fair. IaC engines delete resources that disappear from desired state. Pulumi’s answer is not “trust the author.” It is protect, deliberate unprotect, preview discipline, and optional policy — the same layered approach you would demand for AWS, Azure, or a Kubernetes workload on a home lab that later becomes shared infrastructure.

The failure mode

A typical incident path looks like this:

  1. Someone renames a resource or removes it while “cleaning up.”
  2. pulumi up plans a delete (or a replace, which deletes then creates).
  3. Without a fence, the provider API destroys the cloud or cluster object.

Stateful examples that should almost never be casually deletable: RDS / Azure SQL, persistent volumes, Key Vaults / KMS keys, storage accounts with retention data, production VPCs / VNets.

Mark resources with protect: true

The protect resource option tells the Pulumi engine: if this update would delete (or replace) the resource, fail the deployment instead. Nothing is destroyed until protection is removed on purpose.

TypeScript

import * as aws from "@pulumi/aws";
import * as pulumi from "@pulumi/pulumi";

const config = new pulumi.Config();
const protect = config.getBoolean("protect") ?? false;

const bucket = new aws.s3.Bucket(
  "app-data",
  { forceDestroy: false },
  { protect },
);

Python

import pulumi
from pulumi_azure_native import storage

config = pulumi.Config()
protect = config.get_bool("protect") or False

account = storage.StorageAccount(
    "appdata",
    resource_group_name=resource_group.name,
    sku=storage.SkuArgs(name=storage.SkuName.STANDARD_LRS),
    kind=storage.Kind.STORAGE_V2,
    opts=pulumi.ResourceOptions(protect=protect),
)
Prod pattern: set protect: true in production stack config and leave it false (or unset) for disposable stacks. Same program, different blast radius.

Inheritance on components

Setting protect: true on a component resource propagates to child custom resources. That is useful for Azure resource groups, AWS VPCs packaged as components, or a Kubernetes app component that owns deployments, services, and PVCs — protect the parent when the whole unit must not vanish.

Override a child with protect: false only when that child is intentionally disposable under a protected parent.

Blanket protection with stack transforms

There is no single “protect everything” flag, but a stack transform can set protect on every resource as it registers — ideal for production stacks across AWS and Azure without sprinkling options on each declaration: 

import * as pulumi from "@pulumi/pulumi";

const config = new pulumi.Config();
if (config.getBoolean("protect")) {
  pulumi.runtime.registerStackTransformation((args) => ({
    props: args.props,
    opts: pulumi.mergeOptions(args.opts, { protect: true }),
  }));
}

Unprotect is a two-step, intentional act

To delete a protected resource you must first clear protection:

  1. Set protect: false in code/config and run pulumi up, or run pulumi state unprotect <urn>
  2. Run a following update that actually deletes the resource

That two-step path is what you show reviewers: destruction cannot be a single accidental click after a bad merge.

CLI pulumi state protect caveats

pulumi state protect flips the protect bit in state. If the program does not also set protect: true, the next successful update can clear that bit again. Treat CLI protect as a temporary fence; encode the option in the program or transform for durable policy.

protect vs retainOnDelete

  • protect — block the delete; keep managing the resource in Pulumi.
  • retainOnDelete — allow Pulumi to drop the resource from state without calling the provider delete (orphans the cloud object on purpose).

Use protect for “never destroy by accident.” Use retainOnDelete for “stop managing this without wiping it.”

Second fence: Policy as Code

For orgs that want a central guarantee, CrossGuard / Policy as Code can deny updates that would leave critical resource types unprotected, or deny deletes of tagged production resources. Pair that with mandatory CI previews.

Reviewer checklist (paste into an ADR)

  • Production stacks set protect: true via config or a stack transform
  • Databases, disks, keys, and networking foundations are explicitly protected
  • CI runs pulumi preview on every change; deletes are reviewed
  • Unprotect requires a separate change (or break-glass state unprotect) before destroy
  • Optional: policy pack enforces protect on critical types
  • Same rules apply whether the stack targets AWS, Azure, or Kubernetes

Faulty deletes are a process and defaults problem. Pulumi gives you the same class of control Terraform’s prevent_destroy and CloudFormation deletion policies give — expressed as a first-class resource option you can apply consistently everywhere you already run Pulumi.
