Migration

Import Terraform, CDK, or CloudFormation into Pulumi

Move existing stacks without downtime: convert code, import state, verify a clean preview — and keep protection on until cutover is deliberate.

Adoption stalls when teams hear “rewrite everything.” You do not have to. Pulumi can adopt infrastructure that already exists under Terraform, AWS CDK, or CloudFormation, then keep managing it from the same workflow you use for Azure and Kubernetes stacks.

The safety theme from protect applies here too: imports should land protected so a half-finished conversion cannot unlock a mass delete.

Shared checklist

  • Convert or generate Pulumi code for the stack
  • Import existing resources into Pulumi state (no cloud changes)
  • Run pulumi preview until it shows no changes
  • Keep protect until refactor and review are done
  • Only then unprotect and change resources on purpose
  • Migrate environment by environment (dev → staging → prod)

From Terraform

Two jobs: turn HCL into a Pulumi program, and adopt the live resources from Terraform state.

Convert code

# from a directory of .tf files
pulumi convert --from terraform --language typescript --out ../pulumi-app

Import state

pulumi import --from terraform ./terraform.tfstate
Deletion safety: resources imported from Terraform state are marked protected. That is intentional — you can fix the generated program while the engine refuses accidental deletes. Unprotect only after preview is clean and reviewers agree.

Alternatives when the estate is large: Pulumi Neo for assisted zero-downtime migration, or a state-first path with pulumi-terraform-migrate plus code conversion. You can also keep coexisting with Terraform by reading a .tfstate for references during a gradual move.

From AWS CDK

CDK apps deploy through CloudFormation. Treat each CDK/CloudFormation stack as a migration unit — partial moves are risky while CFN still owns the stack.

  1. Synthesize the CDK app (cdk synth → cdk.out)
  2. Convert with Neo (recommended) or cdk2pulumi against the cloud assembly
  3. Import live resources with Neo or the cdk-importer tool
  4. Confirm pulumi preview is empty before cutting over
# example: generate import mappings / import via cdk-importer
pulumi plugin install tool cdk-importer
pulumi plugin run cdk-importer -- -stack MyCdkStack

pulumi preview   # expect zero changes before you rely on Pulumi alone

If you need a softer landing, the Pulumi CDK adapter can run CDK constructs on the Pulumi engine first; full conversion can wait until the team is ready.

From CloudFormation

Pure CloudFormation stacks carry a complete inventory of logical IDs and physical resources. Neo (and related CFN migration guides) read that inventory, generate Pulumi code, and import without tearing down the stack.

  • Migrate whole stacks, not random subsets still updated by CFN
  • Prove zero drift with preview after import
  • Disable or remove the old CFN update path only after Pulumi owns the resources

Azure and Kubernetes neighbors

Even when the legacy stack is AWS-only CDK or Terraform AWS, the destination can sit beside Azure Pulumi stacks and a home-lab Kubernetes project under one CLI and one review culture. Import does not force a big-bang multi-cloud rewrite — it lets you stop the bleed on the stack you are migrating first.

What “ease” actually means

Ease is not skipping review. Ease is: automated convert + import, protected resources by default, and a preview that proves you changed nothing in the cloud before you change anything. That is the path that survives the same org question as greenfield Pulumi — how do we avoid faulty deletes? — while you leave Terraform, CDK, or CloudFormation behind on purpose.
