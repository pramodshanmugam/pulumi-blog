Why prefer Pulumi over Terraform and CDK

One programming model for AWS deployments, Azure deployments, and a Kubernetes cluster on a home laptop — without trading away the guardrails platform teams demand.

I use Pulumi day to day for AWS, Azure, and Kubernetes on a home laptop. That mix is exactly where tool sprawl hurts: Terraform for one cloud, CDK for another, Helm charts or raw manifests for the cluster, and a different review story for each. Pulumi collapses that into one engine, one preview, and one set of safety options.

Real languages instead of a DSL

Terraform’s HCL is excellent at declaring cloud resources, but loops, conditionals, and shared abstractions stay inside a DSL. AWS CDK brings real languages — then synthesizes to CloudFormation, so the runtime truth is still a template pipeline.

Pulumi programs are ordinary TypeScript, Python, Go, C#, or Java. You get IDEs, packages, tests, and the same control flow you already use for applications. For a home k8s lab that grows into something more serious, being able to share helpers between an Azure stack and a local cluster matters more than another YAML dialect.

One model across AWS, Azure, and Kubernetes

CDK is strongest inside AWS. Terraform covers many providers, but each environment still tends to grow its own module conventions. With Pulumi I keep the same workflow:

  • AWS accounts for production-shaped workloads
  • Azure subscriptions for other services and identity
  • A local Kubernetes cluster for fast iteration

Providers change; pulumi preview, stack config, secrets, and resource options do not. That consistency is the practical reason to prefer it when you are not all-in on a single cloud.

Preview is the first delete fence

Reviewers often fear that “real code” means a bad loop can wipe prod. The first answer is cultural and mechanical: never apply without a reviewed preview. Pulumi’s diff makes creates, updates, replacements, and deletes explicit before anything runs.

Org approval note: treat CI preview on every PR as mandatory. Faulty deletes are almost always “we skipped the diff,” not “the engine invented a delete.”

Delete safety compared with Terraform and CDK

Terraform’s lifecycle { prevent_destroy = true } and CloudFormation / CDK deletion policies are real protections. Pulumi’s equivalent is the protect resource option — and it fits the same mental model platform teams already accept:

  • Mark stateful resources (databases, disks, key vaults, persistent volumes) with protect: true
  • Drive that flag from stack config so prod protects and ephemeral stacks can still be destroyed
  • Use stack transforms to apply protection broadly in production
  • Layer Policy as Code if the org wants a central rule: “critical types may not be unprotected”

The longer walkthrough — including unprotect workflow and reviewer checklist — is in How Pulumi protect stops faulty deletes.

Where Terraform or CDK can still win

Be honest in an ADR. Large Terraform estates with mature modules may not need a rewrite tomorrow. Teams deep in CDK constructs may prefer the Pulumi CDK adapter or a staged import. Preferring Pulumi does not require pretending other tools are empty. It means one engine for multi-cloud plus Kubernetes, with delete controls that survive org review.

Bottom line

I prefer Pulumi because the same program style covers AWS, Azure, and home-lab Kubernetes, the preview is trustworthy, and protect gives a clear answer when someone asks: “What stops a faulty delete?” That question — more than syntax preference — is usually what decides approval.
